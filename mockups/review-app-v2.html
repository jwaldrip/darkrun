checkpoint decision logic   The two buttons aren't static. Clean (no feedback, or only nits) → Approve is primary, Request changes is the quiet secondary. The moment you leave any annotation tagged high or blocker anywhere on the station, Approve darkens (you can't cleanly approve over open blockers) and Request changes becomes primary. Request changes also takes one global station note (above) which ships alongside all the per-artifact annotations — that single submit kicks the run into the feedback loop. So severity does the steering: nits don't block, high/blocker flip the primary. Visibility: this whole bar only renders at an active review or the final checkpoint (the gate is open / blocking). Mid-manufacture there's nothing to decide, so it's hidden — feedback still queues, and the inbox stays reachable from the header. See section E for the mid-work state.
what changed   ① Stations are the top stepper (done=check, current=cyan diamond w/ glow, pending=outlined number; connectors color-coded; amber feedback dot) — this is the assembly line. ② Phases demoted to a compact subheader scoped to the current station. ③ Units carry dependencies inline (no separate DAG panel). ④ One checkpoint control set. Header is sticky — scroll the window to see it pin.
one method, many lines   The station strip is per-factory — these six (Frame→Harden) are the software factory's line, read from its FACTORY.md: stations:. A different factory renders a different line. The phase subheader is universal — every station in every factory walks the same Spec→Review→Manufacture→Audit→Reflect→Checkpoint machine (your call: the phase-machine stance is correct, stays fixed). So the strip must be data-driven from the run's factory, not a hardcoded six.
naming to settle   Station Specify vs phase spec sit close. Worth a rename so the two levels never blur — e.g. phases as verbs (draft → review → make → audit → reflect → gate)? Your call.

Launching the harness boots the darkrun MCP engine, which advertises itself to ~/.darkrun/. The desktop is already on this run — it links the instant the engine appears. No second step.

autonomous by default   darkrun moves the permission boundary from per-tool-call to per-station-checkpoint — the agent runs hot inside a station, you gate at the review. So the generated command launches the harness with prompts bypassed, with a toggle to dial back to cautious (per-tool prompts) per project. Per-harness flag from the registry: claude --dangerously-skip-permissions · codex --full-auto (or --dangerously-bypass-approvals-and-sandbox) · gemini --yolo · Cursor/Windsurf/Kiro expose it as an auto-run/turbo setting, not a flag. (exact flags per harness — confirm against current CLI versions.)
one step, not two   You're already in the desktop with the run selected, so there's nothing to show — opening the run was the selection. The single command just starts an engine for the project; it boots the MCP server (plugin already configured), which writes its descriptor to ~/.darkrun/<slug>/, and the desktop auto-connects. darkrun_show is gone from the user path — it's only ever an agent-side "raise the window" convenience.
